LLM Cost Optimization and Token Management

As language model adoption scales from prototype to production, inference costs become a primary engineering concern. At scale, poorly optimized LLM usage can cost hundreds of thousands of dollars per month in API fees. Cost optimization is a first-class engineering discipline in AI infrastructure.

Key Cost Drivers:
- Input tokens: Every token in the prompt (system prompt + context + user query) is billed.
- Output tokens: Generated response tokens — typically 3-5x more expensive than input tokens.
- Model tier: GPT-4 can be 50-100x more expensive per token than GPT-3.5 or open-source models.
- Request rate: Wasted retries from rate limit errors increase costs.

Optimization Strategies:

1. Model Routing / Intelligent Routing:
Route simple queries to cheaper models (Llama-3-8B, GPT-3.5) and complex queries to powerful models (GPT-4, Claude-3-Opus). A classifier or heuristic determines query complexity. Can reduce costs by 60-80% with minimal quality impact.

2. Prompt Compression:
- Remove redundant whitespace, verbose instructions, and unnecessary examples.
- Use LLMLingua or Selective Context to compress prompts via perplexity-based token pruning.
- Token compression can reduce prompt length by 3-5x with <5% performance degradation.

3. Caching:
- Exact match caching: Cache identical prompts (high cache-hit for high-traffic chatbots).
- Semantic caching (e.g., Redis with vector similarity): Cache semantically similar queries.
- KV cache reuse: Some providers (Anthropic, Google) offer prefix caching for repeated system prompts.

4. Context Window Management:
- Truncate retrieved context to only the top-k most relevant chunks.
- Use map-reduce for long documents rather than stuffing everything into context.
- Compress conversation history using summarization rather than passing full raw turns.

5. Output Length Control:
Set max_tokens limits. Instruct the model to be concise. Shorter answers = lower cost + faster first token latency.

6. Batching:
Use the Batch API (OpenAI, Anthropic) for offline/non-real-time jobs. Costs typically 50% less than real-time API calls.

Token Efficiency Metric:
A useful SRE signal: quality_per_output_token = answer_quality_score / log(1 + output_tokens). Tracks whether answer quality improves proportionally as output length increases. A declining ratio indicates the model is becoming more verbose without being more accurate.
